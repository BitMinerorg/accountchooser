Prerequisites

First we assume that your website requires users to enter an email address and password to login. Visit the Google APIs Console Configuration to activate the GITkit API, and get the generated JavaScript snippet which includes the config parameters, API keys, etc. The Google API Console is a tool that lets you manage the requests your applications make to Google APIs. The Console lets you:

  • Choose which APIs to use in your project
  • Share your project with other people or a team
  • Define traffic controls to limit who and how frequently (QPS) your application makes API requests
  • See traffic reports of your project's API usage over time
  • Manage your keys for correct request identification

For more information of general Google APIs Console usage, see its help page.

API Integration Options

GITkit provides several components to simplify the work needed for RPs.

  • GITkit API and Service handles the details of the federated login protocols between RP and Identity Providers (IDP).
  • JavaScript widgets for login and account signup can be embedded into your web pages easily.
  • GITkit client libraries communicate with the Google Identity Toolkit using the GITkit API, and drive the UI flow of the JavaScript widgets.

There are a few different options for how to modify your server to use the GITKit widget:

  • Use Account Chooser: follow this guide to use the newest login UI component that supports both federated login and legacy login.
  • Direct API access: Since there is only one simple API to call from your server, you may choose to integrate directly with the API.
  • Server-side libraries (called GITkit client libraries): We have also provided libraries for certain languages/frameworks that may simplify the server side integration, especially for new sites.
  • Sample apps: The last option is to download a sample app that is already integrated with GITKit, and then modify it to build a new site. Currently we have three sample sites that are provided with different GITkit packages and RP scenarios.

Optional Tasks

The following is a list of optional changes you may make to your website to improve the user experience for federated login.

  • Modify your new account creation page, so that when creating a federated account, there is no need for the user to provide a password.
    Your new account servlet can detect the account type (federated or legacy) by looking at the session state. Your new account page can also import the attributes that are exchanged with IDPs by GITkit, such as email address, first name, last name, etc, and pre-fill the account creation form so that it is easy for users to signup.
  • Modify your account management page that allows a user to change their email address.
    If the user has an OpenID enabled account and changes their email address to a non-OpenID enabled account, then they need a way to set a password on the account. You can either ask them to provide a password at the same time they change their email, or you can email them a password reset link if your site already has the code for sending such a link.
  • Modify your account management page to remove the option for a user to modify their password if their account is configured to use federated login.
    If you provide an installed application for your users (such as an Android or iPhone app), then it probably is configured today to ask the user for their email address and password. That will not work for federated login users because they do not have a password on your site. For these users you should replace the "change password" link on your website to instead be called "View account password." If the user visits that page you should display a randomly generated password for the user, and then save that password into the user record in your database that is traditionally used to store a password. That will then allow the user to login to your Android/iPhone/etc. app using that password. You can then modify your Android/iPhone app to point users to this page so that they can view that password. In the future the Google Identity Toolkit will provide additional options to enable Android and iPhone apps to also use federated login and avoid the need for this generated password.
